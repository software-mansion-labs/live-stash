# Modes

## Client mode

In client mode stashed state is saved to the browser memory through `Phoenix.LiveView.push_event/3`.

### Stash structure

State is saved in the following form after `LiveStash.stash_assigns/2` is called:

```javascript
stashedState:{
    "assigns":{
        "encoded_key_1": "signed_or_encrypted_value1",
        "encoded_key_2": "signed_or_encrypted_value2",
    },
    "keys": "signed_or_encrypted_list_of_stashed_keys"
}
```

### State recovery

Updated socket is returned from `LiveStash.recover_state/1` only if **every** key-value pair from the signed list of stashed keys was succesfully recovered.

### Reseting the stash

Stash is cleared always after a LiveView using **client** mode is rendered for the first time. You can also do it manually with `LiveStash.reset_stash/1`. Obviously refreshing the browser tab clears the state too.

## Server mode

### Stash structure

#### ETS

State is saved to ETS in the following form after `LiveStash.stash_assigns/2`is called:

```elixir
@type t ::
record(:state,
id: term(),
pid: pid(),
delete_at: integer(),
ttl: integer(),
state: map()
)
```

**id** - UUID generated by LiveStash per LiveView concatenated with the [secret](./configuration.md) and hashed

**pid** - PID of the LiveView process currently owning the stash

**delete_at** - time of creation + ttl

**ttl** - time to live of the record

**state** - your stashed state

#### Browser

In server mode **node hint** and **stash id** are saved in browser memory.

**node hint** - last node that the client was connected to, it is used in a clustered environment to optimize the retrieval of the state

**stash id** - UUID generated generated by LiveStash per LiveView used to create the ETS record id
